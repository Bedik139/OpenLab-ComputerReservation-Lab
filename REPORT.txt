PROJECT OVERVIEW
----------------

Handlebars, Mongoose (MongoDB), and Bootstrap. The project follows the
mc3-debttracker folder structure: index.js at the root as the entry point,
all source code inside src/ (models, routes, controllers, middleware,
helpers, views), and static assets in public/.

3. HOW AUTHENTICATION WORKS
================================================================================

- User accounts are stored in MongoDB (users collection)
- Passwords are hashed with bcrypt (pre-save hook on User model)
- Login validates email + password via bcrypt.compare()
- On success, user object (without password) is stored in req.session.user
- Sessions are persisted in MongoDB via connect-mongo
- Registration creates a new User document (password auto-hashed)
- Duplicate emails and student IDs are rejected (unique index)
- After registration, the user is auto-logged in and redirected to dashboard
- Logging out destroys the session and clears the cookie
- Password changes use comparePassword() + pre-save hook for new hash

  - Edit their own profile (no college field, no delete account)
  - "Available Slots" nav link is hidden for technicians

Node.js server.

Key changes to js/app.js:
  - authenticateUser() now sends POST /api/login with JSON body
    {email, password} and receives a session cookie on success.
  - registerAccount() now sends POST /api/register with JSON body
    containing all registration fields.
  - getReservations() now sends GET /api/reservations and parses
    the JSON response to populate the reservations page.
  - saveReservation() now sends POST /api/reservations with the
    reservation object as JSON.
  - cancelReservation() sends DELETE /api/reservations/:id.
  - editReservation() sends PUT /api/reservations/:id with updated fields.
  - getAllReservations() (technician) sends GET /api/reservations/all.
  - updateProfile() sends PUT /api/profile with updated user fields.

  - Walk-in endpoints: POST /api/walkin, GET /api/walkin,
    DELETE /api/walkin/:id.
  - All fetch calls include { credentials: 'include' } to send session
    cookies with every request.

C. COOKIES AND SESSIONS
------------------------
Implemented server-side session management using express-session with
a MongoDB session store (connect-mongo).

Setup (in index.js — the main entry point):
  - express-session configured with:
      secret: environment variable SESSION_SECRET
      resave: false
      saveUninitialized: false
      cookie: { maxAge: 1000*60*60*24 } (24-hour sessions)
      store: MongoStore.create({ mongoUrl: MONGO_URI })
  - On successful login, the user object (without password) is stored
    in req.session.user.
  - authMiddleware checks req.session.user on protected routes and
    redirects to /login if absent.
  - techMiddleware additionally checks req.session.user.accountType
    === 'technician' for technician-only routes.
  - Logout destroys the session via req.session.destroy() and clears
    the cookie.
  - "Remember Me" checkbox sets a longer maxAge (21 days) on the
    session cookie.

Session-protected routes:
  GET  /dashboard          → authMiddleware
  GET  /reservations       → authMiddleware
  GET  /reserve            → authMiddleware
  GET  /profile            → authMiddleware
  PUT  /api/profile        → authMiddleware
  POST /api/reservations   → authMiddleware
  GET  /walkin             → authMiddleware + techMiddleware
  POST /api/walkin         → authMiddleware + techMiddleware
